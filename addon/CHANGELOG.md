## 1.2.175
- **Fix: in-UI channel badge missing on dev/beta — `addon/.channel` wasn't being copied into the container image** — v1.2.174 shipped the channel-identity plumbing (`addon/.channel` per branch + `/api/channel` endpoint + `initChannelBadge()` injection of the orange/blue outlined pill into the nav and `[DEV]`/`[BETA]` document title prefix), and the corner-stamped icon/logo + repository.yaml name suffix were correctly differentiating in HA's repo list and supervisor list. But on the dev addon at runtime, the orange "DEV" pill never appeared in the UI and `[DEV]` never prefixed the tab title — even though v1.2.174 was confirmed installed and the D-stamped icon next to "Home" was visible. Network tab showed `/api/channel` returning `{"channel": "main"}` despite `addon/.channel` containing `dev` on the dev branch in git. Root cause: `addon/Dockerfile:19` copies the addon source into the image with the glob `COPY *.py *.html *.js *.png requirements.txt ./` — that pattern excludes dotfiles like `.channel` (and would also exclude a non-dotfile `channel` since it has no extension matching the glob list). The file existed in the repo but never made it into the running container, so `_get_channel()` (`webserver.py:_get_channel`) hit its file-not-found branch and returned the fallback "main"; `initChannelBadge()` then took the early-return path for `channel === 'main'` and rendered nothing. The static icon worked because `*.png` in the existing glob already matched `icon.png`/`logo.png`. Fix: add an explicit second COPY directive `COPY .channel ./` immediately after the existing COPY line. Kept the dotfile name (it's metadata, conventional). One-line Dockerfile change, no Python or JS edit needed; v1.2.175 build will rebuild the image with the file present, `_get_channel()` will read `dev`/`beta`/`main` correctly per branch, and the badge + title prefix will render. Also a useful precedent: dotfiles in `addon/` need explicit COPY directives — the existing glob pattern won't pick them up, and silently falling back to a default value (as `_get_channel()` does) makes the breakage invisible until you check the network tab. (The same gotcha bit `_get_addon_version()` in v1.2.172, which is why that version had to switch to reading from a build-arg ENV instead of `config.yaml` — but `.channel` is per-branch and we don't want it in build.yaml, so the explicit COPY is the right fix here.)

## 1.2.174
- **Channel identity per branch: distinct repo name + addon icon/logo on dev/beta/main + in-UI channel badge** — addresses the user's "I can't tell dev from beta from main when both are installed in HA" pain. Two layers of differentiation now ship together. **Layer 1 — channel files in git:** dev/beta/main each carry a different `repository.yaml` `name` ("Circadian Light by HomeGlo (Dev)" / "(Beta)" / unsuffixed for stable), a different `addon/icon.png` and `addon/logo.png` (corner-stamped "D" or "B" badge generated via Pillow from the original logo, white circle with navy letter overlapping the gradient ring; stable keeps the original unbadged), and a new `addon/.channel` text file containing `dev` / `beta` / `main`. Color-only differentiation rejected because (a) HA sidebar's `panel_icon` is MDI (always monochrome) and (b) HA desaturates the addon logo to grayscale when the addon is stopped — corner letter stamp survives both. Per-branch identity protected via `.gitattributes merge=ours` on the four files (repository.yaml, addon/icon.png, addon/logo.png, addon/.channel) so future dev → beta → main merges propagate code and version normally but never overwrite identity. Requires `git config merge.ours.driver true` once per clone (built-in `ours` driver isn't registered by default); new `scripts/setup-git.sh` one-liner does this and is documented for fresh-clone onboarding. **Layer 2 — in-UI channel badge:** new `/api/channel` endpoint reads `addon/.channel` once at startup and exposes the value; `shared.js` fetches on every page load and, for dev or beta only, injects an outlined pill ("DEV" in `#feac60` orange / "BETA" in `#5fa8ff` blue) into the nav next to the brand link, plus prefixes the document title with `[DEV]` / `[BETA]` so the browser tab differentiates too. Outlined-pill style chosen over solid-fill because it's visible at a glance but doesn't compete with content; main shows nothing (stable should look "normal"). Badge code is one self-contained function `initChannelBadge()` that gracefully no-ops if the API fetch fails (running outside HA, network error, etc.). The plumbing avoids touching `addon/config.yaml` (which propagates version through merges and would force a partial-merge gymnastic) by routing identity through `addon/.channel` instead — clean separation between "what the addon IS" (per-branch) and "what version it's at" (cross-branch).

## 1.2.171
- **"Report" button on every device row in the Add control modal — one-click submission to homeglo for unsupported-device curation** — first half of the external-registry vision (project_camera_triggers.md). Today, devices not in the allowlist (CAMERA_MODELS/MOTION_SENSOR_MODELS/CONTACT_SENSOR_MODELS in switches.py) require us to manually curate each manufacturer/model and ship a new addon version. Users can still add them via the manual flow (search → pick → assign triggers), but we have no signal about which devices are in the field, so we can't prioritize what to allowlist next. This change adds the feedback channel: every device card in the Add control modal now has a small "Report" button (top-right, opacity 0.5 default → 1.0 on hover so it doesn't compete with the device-pick affordance, click stops propagation so it doesn't also fire `selectDevice`). Click sends manufacturer + model + integration + sw_version + addon_version + full binary_sensors list (entity_id + device_class for each) to a Cloudflare Worker at `homeglo-device-reports.rweisbein.workers.dev`, which files it as a GitHub issue in `rweisbein/device-requests` (private repo) using a fine-grained PAT scoped to issues:write on that one repo. Backend implementation: new `HOMEGLO_REPORT_WEBHOOK_URL` constant near top of webserver.py + `_get_addon_version()` helper that reads `version:` from config.yaml lazily and caches it (avoids three-place version bumps — config.yaml stays the single source of truth) + new `report_device` endpoint at `/api/devices/report` that pulls the device from `client.device_registry`, walks `client.entity_registry` to collect all binary_sensor entities for that device_id with their `device_class`, builds the JSON payload, and POSTs to the webhook with a 10s timeout + structured error responses (502 for webhook errors, 504 for timeout, 503 if WS client not connected). Frontend: `reportDevice(deviceId, btn)` JS function with optimistic button state ("Sending..." → "Thanks!" or "Failed", reverts after 2.5s); CSS rules for `.acp-report-btn` (small, muted, hover-fade in). The webhook itself is a 70-line CF Worker (deployed separately via Cloudflare dashboard) that validates payload shape, builds a markdown issue body with manufacturer/model/integration/sw_version/addon_version + bulleted binary_sensors list with device_class, and POSTs to `api.github.com/repos/rweisbein/device-requests/issues` with a `device-request` label. CORS headers permissive (Allow-Origin: *) so the addon iframe can POST cross-origin. End-to-end verified: console fetch from example.com to the worker URL filed issue #1 in the device-requests repo successfully. Net effect: a user opens Add control, sees their device in the list, clicks "Report" — within seconds, an issue lands in our triage queue with everything we need to add an allowlist entry. No GitHub account required from the user, no JSON to copy-paste, no email follow-up. Manual-add flow unchanged — Report is purely a signal channel, not a substitute for adding the device. Future: extend Report button placement to the controls list itself (for inactive/legacy entries that didn't make it into the allowlist), and replace the bundled allowlist with a homeglo-hosted models.json that the addon fetches on startup, so new device support ships without an addon update.

## 1.2.128
- **Phase slider now narrows its range dynamically so the shifted midpoint stays inside the phase window** — fixes the 1.2.125-reported bug where setting bed=12:48p (cursor just past descend_start with `bed_brightness ≠ 50%`) rendered the curve as if bed were ~2a. - **Faster home page refresh**: Default refresh interval reduced from 10s to 3s. Existing users are migrated automatically. Switch presses and motion triggers now appear on the home page almost immediately.
- **Suppress access logging**: HTTP request logs from polling disabled to keep logs clean at higher poll rates.

## 1.0.56
- **Unified process architecture**: Webserver now runs embedded in the main process instead of as a separate background process. This is a structural change (Phase 1 of process merge) — no behavior changes yet, but lays the groundwork for eliminating redundant computation and WebSocket connections in the polling endpoints.

## 1.0.33
- **Direct weather outdoor brightness formula**: Replaced log-compressed lux estimation with a direct `elevation_factor × condition_multiplier` formula for weather and angle sources. This dramatically improves differentiation between weather conditions — sunny vs rainy spread goes from ~12% to ~46%. The lux sensor path is unchanged.
- **Configurable weather condition strengths**: Settings > Outdoor Brightness now shows slider controls for each weather condition group (Sunny, Partly cloudy, Cloudy, Rainy, Snow, Fog, Pouring, Storm) when source is Weather or Sun Angle. Adjustments are saved as `weather_condition_map` in config.
- **Latitude-aware elevation scaling**: Max summer solstice elevation is computed from your Home Assistant latitude, so the elevation factor correctly reaches 1.0 at your location's peak sun angle rather than using a fixed reference.

## 1.0.32
- **Cancellable post-command nudge**: After every light command (switch press, motion trigger, step, color change), the same values are re-sent 1 second later to catch dropped Zigbee commands. Per-area tracking means a new action cancels only that area's pending nudge. Replaces the old third-step retry with broader coverage across all command types. Configurable in Settings > Zigbee Improvements (0 = disabled).
- **New Zigbee Improvements settings section**: Two-step delay and post-command nudge settings are now grouped under their own section, separate from Feedback Cues.

## 1.0.31
- **Fix Hue motion sensors being disabled**: Switch entity collection for area on/off now skips entities with `entity_category` of "config" or "diagnostic". Previously, Hue motion sensor enable/disable switches (`switch.*_sensor_enabled`) were included as area switch entities, causing them to be turned off whenever the area turned off — disabling the motion sensors on the Hue hub.

## 1.0.30

- **Cheat sheet visual improvements**: Button name and icon are now vertically centered across all action rows for that button. Stronger visual separators between button sections.
- **Sticky column headers**: Controls page table headers stay visible below the nav bar when scrolling.
- **Filter coordinator from area lights**: Service devices (coordinators, bridges) no longer appear in the area detail light list.

Previously, any brightness override (including negative/dimming) blocked auto-off; now only positive overrides (user brightened) prevent it.

## 1.0.29

- **Reach groups for step/bright**: Step and brightness commands on multi-area switches compute values without sending, then dispatch via reach groups when all areas match. A 3-area step_down_3 drops from 18 Zigbee commands to 2.
